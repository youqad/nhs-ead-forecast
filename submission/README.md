# Submission

Leakage-safe NHS-EAD forecast candidate using a fixed horizon split selected on
assessment-style winter validation. Prizes 1to5 and 6to10 are scored
independently, so the default configuration uses the strongest validated system
for each block.

## Setup

```bash
uv run --with-requirements submission/requirements.txt python -V
```

## Run

Single official 10-day forecast (the contest's 1-hour rule applies here):

```bash
uv run python -m submission.run_forecast --config submission/config/default.yaml --origin 2025-10-01
```

Full assessment matrix (all 173 origins). By default this recomputes every
per-origin forecast; pass `--reuse-existing` only when the cached forecast CSVs
are known to have been generated by the current config.

```bash
uv run python -m submission.scripts.run_final_forecast --config submission/config/default.yaml
```

Validate deliverables:

```bash
uv run python -m submission.scripts.validate_submission --config submission/config/default.yaml
```

## Tests

```bash
uv run --with-requirements submission/requirements.txt --with pytest pytest -q
```

## Outputs

- `submission/pred_matrix.csv` — 173 rows × (forecast_id + day_1..day_10)
- `submission/report.pdf` — ≤1000 words
- `submission/report.md` — report source

## Results

The fixed horizon split in `submission/config/default.yaml` is the submitted
model:

- days 1-5: Chronos + NB-INGARCH-AQ + seasonal residual correction
- days 6-10: NB-INGARCH-AQ

Assessment-style validation motivating the split:

| Horizon block | Selected model | MSE |
| --- | --- | ---: |
| days 1-5 | Chronos + NB-INGARCH-AQ + seasonal residual correction | 0.1157 |
| days 6-10 | NB-INGARCH-AQ | 0.1293 |
| combined | fixed horizon split | 0.2450 |

The submitted `pred_matrix.csv` is frozen in the official 11-column format:
`forecast_id, day_1, ..., day_10`.
